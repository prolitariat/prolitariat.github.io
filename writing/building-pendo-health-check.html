The Problem

If you've ever managed a Pendo implementation, you know the drill. Guides stop showing and nobody knows why. Visitor data looks wrong but the snippet seems fine. A CSP policy silently blocks the agent and nobody notices for weeks. Every time, you're digging through browser consoles, checking network tabs, and running pendo.validateInstall() by hand.

I've spent years as a practitioner inside Pendo — first as Sr. Program Manager for Digital Adoption at Jamf, now as a Product Manager at Splunk working on product analytics and in-app content. The job is the same at both: connect data systems, acquire and analyze data, then create in-app content that moves needles. And the debugging workflow was always the same too: open DevTools, poke at window.pendo, check half a dozen things, try to remember what the healthy state looks like.

The Pendo Health Check started as a question: what if all of that was one click?

Starting From Zero

I built this the same way I built my portfolio site — through conversation with Claude in Cowork mode. No IDE. No terminal. Just describing what I wanted and iterating on the results. The entire extension is vanilla JavaScript with zero external dependencies — no npm, no build step, no bundler. You clone the repo, load it unpacked in Chrome, and it works.

The architecture decision to stay dependency-free wasn't accidental. Chrome extensions with complex build systems create friction for contributors and make debugging harder. When the entire extension is three files you can read top-to-bottom (popup.html, popup.js, background.js), the barrier to understanding it drops to zero.

The first commit gave me a working popup with five basic checks. It could detect whether Pendo was loaded, whether it was initialized, and whether visitor and account IDs were present. Useful, but barely scratching the surface of what goes wrong in real implementations.

From 5 Checks to 13

The real value emerged through iteration. Each new check came from a real scenario I'd hit while managing Pendo implementations:

Dual Instance Detection (Check #6) — one of the most subtle and destructive Pendo bugs. When two copies of the agent load on the same page (common with tag managers and SPAs), they fight over state. The check counts <script> tags and inspects the pendo namespace for conflicts.

Network Request Validation (Check #10) — goes beyond "is the agent loaded" to "is it actually talking to the server." Uses performance.getEntriesByType('resource') to find Pendo traffic, then checks for the CORS fingerprint (transferSize === 0 and responseStatus === 0) that indicates a blocked request.

Feature Flag Detection (Check #11) — inspects pendo.getOptions() for non-default configuration, which often explains unexpected behavior that basic checks miss.

Ad Blocker Detection (Check #12) — ad blockers don't just block ads. Many filter lists target analytics scripts, including Pendo's CDN and data endpoints. This check detects whether an ad blocker is interfering with agent loading or data transmission.

Consent Platform Detection (Check #13) — CMP and GDPR consent platforms can delay or prevent Pendo initialization entirely. The check scans for major consent management platforms and flags when consent state may be blocking the agent.

The Setup Assistant

Health checks tell you what's wrong right now. But Pendo problems often stem from how the agent was installed in the first place — the wrong snippet, missing async attributes, CSP policies that block the CDN.

The Setup Assistant is a full implementation audit that runs seven analysis sections: framework detection (React, Angular, Vue, Svelte, Next.js), snippet analysis, initialization method and timing, Content Security Policy compatibility with per-directive fix instructions, CNAME host detection, CMP/GDPR consent platform detection across six major platforms, and metadata field validation with PII scanning.

The CSP analysis alone was a significant engineering challenge. The extension inspects both HTTP headers (via performance.getEntriesByType) and <meta> tags, then checks each relevant directive (script-src, connect-src, img-src, style-src, child-src) to determine whether Pendo's domains are permitted. When they're not, it generates ready-to-paste fix instructions with auto-detected subscription IDs.

Each version wasn't just new features — it was refinement based on real usage. The clipboard system was completely rewritten in v1.3.4 because the original output wasn't actionable enough. Every check now includes context-aware remediation text, and CSP violations include per-directive fix instructions. Reports are structured as plain text suitable for pasting into Slack, Jira, or email.

The QA Test Harness

Testing a Chrome extension that diagnoses another tool's state is inherently tricky. You need a page with Pendo in various states — healthy, broken, CSP-blocked, waiting for consent, behind a CNAME, ad-blocked, partially initialized, running inside a React SPA.

The test harness (test-harness.html) is a self-contained HTML page with 8 preset scenarios and granular toggle controls for every mock dimension: agent state, identity/metadata, guides, network/hosting, CSP modes, CMP/GDPR platforms, and framework detection. You select a preset, click the extension icon, and verify the output.

It mocks window.pendo, performance.getEntriesByType, CMP globals, framework globals, and CSP meta tags — everything the extension inspects. This made it possible to regression test all 13 checks across 8 scenarios without needing access to live Pendo implementations in various states of disrepair.

Accessibility and Craft

Chrome extension popups are notoriously inaccessible. They're small, they use custom styling, and most developers treat them as disposable UI. I wanted to do better.

The extension passes WCAG 2.1 AA with 51 color combinations tested. Every interactive element has focus-visible rings, the tab bar uses proper role="tablist" / role="tab" / role="tabpanel" ARIA attributes, and the first-run tour supports full keyboard navigation — arrow keys to advance/retreat, Escape to skip, Enter to advance.

The first-run tour itself uses a spotlight overlay (box-shadow cutout) with positioned tooltips that walk through each tab and the copy button. It runs once on first install and persists via chrome.storage.local.

The Analytics Dashboard

The extension collects anonymous usage telemetry — popup opens, tab switches, copy report actions, feedback submissions — and ships it to a Cloudflare Workers backend. No PII, no cookies, no personally identifiable data.

The companion analytics dashboard at noskoviak.net/phc visualizes this data in real time: total events, Pendo detection rates, average health scores, version distribution, and daily activity trends. It auto-detects the time range and updates KPIs dynamically.

Building a privacy-first analytics system for a diagnostic tool that itself analyzes analytics tools has a certain recursive elegance to it.

What I Learned

Building a Chrome extension through AI-assisted pair programming taught me a few things that surprised me:

Zero dependencies is a feature. The extension loads instantly, the source code is readable by anyone, and contributions don't require installing a toolchain. When a contributor can understand the entire codebase by reading three files, pull requests get reviewed faster and bugs get found quicker.

Accessibility in small UIs matters more, not less. Extension popups are where people go when they're already frustrated — something isn't working and they need answers. Making that experience keyboard-navigable and screen-reader-friendly is a basic respect issue.

AI pair programming favors fast iteration over planning. I didn't write a spec for this extension. I described what I wanted, looked at the result, and described what needed to change. Sixty-one commits in a few days, each one a small improvement. The cost of trying something wrong is minutes, not hours. That changes how you build.

The best tool is the one that gives you answers in one click. Every check in the extension existed as a manual DevTools procedure before. The value isn't in the diagnostics themselves — it's in removing the friction between "I wonder if Pendo is working" and having a clear answer.

The extension is open source under MIT license. You can install it from the Chrome Web Store, or clone the repo and load it unpacked. Contributions are welcome — the codebase is designed to be extended by anyone comfortable with vanilla JavaScript.
